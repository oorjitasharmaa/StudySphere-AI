🔍 Inverted Index – The Core Data Structure
Forward vs. Inverted Index
Building an Inverted Index
1. Token Sequencing – generate (term, docID) pairs for each document
2. Sorting – first lexicographically by term, then by docID
3. Merging – combine duplicate term-docID pairs, compute term frequencies (TF) and document frequencies (DF)
4. Storing – split into two files:
Dictionary – sorted list of terms, each with DF, collection frequency (CF), and pointer to postings list
Postings – ordered list of docIDs (and optionally TF, positions)
Example Dictionary EntryTechnique Goal Example
Lemmatization Reduce inflectional forms to dictionary
head (lemma)am, are, is → be; cars → car
Stemming Heuristic truncation of suffixes institution, institutional → institut
Index Type Mapping
Forward Index Document → terms (e.g., Doc 1: Bruce loves programming)
Inverted Index Term → list of document IDs (e.g., programming → [1, 3, 4])Query Processing (Boolean Retrieval)
AND query (Alfred AND Bruce):
1. Retrieve postings for each term from dictionary
2. Merge postings by intersecting sorted lists (linear scan, O(|p₁|+|p₂|))
3. Result: docIDs present in both lists
Optimization Tips
Store document frequency (df) with each term; process query terms in increasing df order (smallest
posting first)
For complex Boolean expressions, estimate OR-group sizes and process in ascending size
Skip Pointers – shortcuts inside postings lists to accelerate merging
Ideal number of skips ≈ √P (where P = posting length)
Trade-off: more skips = shorter skip spans but higher storage/comparison cost

Basic Representation
For each term, a sorted list of docIDs (ascending)
Example for "Batman": 2 8 37 54
Memory Required per Token (Intermediate Stage)
Total: ≈ 28–88 bytes per (term, docID) pair
A modest collection such as RCV1 (≈ 100M tokens) would need 2.8–8.8 GB of RAM just for raw pairs – clearly infeasible for a
single machine.
The "Intermediate-Pairs" Problem
While indexing, documents are parsed one at a time, producing term–docID pairs
The final postings for a term are incomplete until the entire collection is processed
Keeping all pairs in memory for sorting is only possible for very small collections; otherwise primary memory
exhausts
🔄 External Sorting for Large Collections
When primary memory cannot hold all intermediate pairs, external-memory algorithms are employed.
Blocked Sort-Based Indexing (BSBI)
Assumptions: Dictionary kept in memory; term-to-termID mapping required for compactness. Using raw terms in
intermediate files would make them huge and slow to merge.
Single-Pass In-Memory Indexing (SPIMI)Approach Space per Term Notes
Fixed-width entry 28–88 bytes Term padded to 20–80 bytes, plus 4
bytes df, plus 4 bytes pointer. Wasted
space: average English term ≈ 8
characters
String-pool alternative ≈ 19 bytes All terms concatenated in single byte
array; 3-byte pointer per term. ~4×
reduction vs. fixed-width

🔎 Retrieval Models – From Boolean to Positional
Boolean Retrieval
Dictionary + postings allow Boolean queries (AND, OR, NOT)
Example: "Alfred Bruce" → intersect postings → result {2, 4, 8, 37, 54}
Limitation: Only records presence/absence; no term importance, no frequency, no positional information
Beyond Boolean – Positional Indexing
Posting list entries can be enriched with:
Enables:
Phrase queries (exact order, distance = 1)
Proximity queries (ordered/unordered window)
Ranking (TF-IDF, BM25)

Heaps' Law
Describes vocabulary growth with corpus size

Standard Analyzer
from java.io import StringReader
from org.apache.lucene.analysis.standard import StandardAnalyzer
from org.apache.lucene.analysis.tokenattributes import CharTermAttribute
text = "This is how we do it."
analyzer = StandardAnalyzer()
stream = analyzer.tokenStream("", StringReader(text))
stream.reset()
tokens = []
while stream.incrementToken():
    tokens.append(stream.getAttribute(CharTermAttribute.class).toString())
print(tokens)   # ['this', 'is', 'how', 'we', 'do', 'it']

🎯 Putting it All Together
📚 Core IR Concepts
Information Retrieval is the process of locating unstructured textual documents that satisfy a user’s information need.
Document representation follows a two ‑ step pipeline:
1. Text processing – tokenisation, case ‑ folding, stop ‑ word removal, stemming/lemmatization.
2. Indexing – building an inverted index (dictionary + posting lists).
Retrieval models evolve from simple Boolean logic to ranked approaches such as TF ‑ IDF, BM25, language models,
and neural embeddings (BERT/LLMs).
Evaluation relies on set ‑ based measures (precision, recall) and rank ‑ aware metrics (MAP, nDCG, MRR) applied to a
common Cranfield ‑ style test collection.
🗂 Inverted Index Construction & Storage
Skip pointers are placed roughly at √P intervals (P = posting length) to accelerate Boolean merging.
🛠 PyLucene Workflow
1. Initialize the JVM (lucene.initVM()).
2. Create a StandardAnalyzer (or language ‑ specific analyzer) to tokenise text.
3. Configure an IndexWriter with an FSDirectory (or RAMDirectory).
4. Build Document objects using appropriate field types:
TextField – indexed + analyzed, optionally stored.
StringField – indexed + untokenised (ideal for IDs, URLs).

🌍 Multilingual & Non ‑ English Analysis
Analyzers are language ‑ specific (e.g., GermanAnalyzer, EnglishAnalyzer, TamilAnalyzer).
For mixed ‑ language collections, use a PerFieldAnalyzerWrapper to assign each field its proper analyzer.
Indian ‑ language corpora (Bengali, Hindi, Tamil, Telugu) benefit from Unicode ‑ aware tokenisers and
language ‑ appropriate stop ‑ word lists.
🔎 Index Exploration with Luke
Luke visualises the dictionary, term statistics, stored fields, and postings.
Compatibility: the Luke version must match the Lucene index version (e.g., Luke 10 ↔ Lucene 10).
Useful checks:
1. numDocs() vs. maxDoc() – live vs. total slots.
2. Token stream displayed for a field reveals the effect of the chosen analyzer (stop ‑ word removal,
stemming).
📈 Evaluation & Benchmarking
Adopt a fixed collection, fixed query set, fixed relevance judgments to ensure fair comparison.
Apply statistical tests (t ‑ test, paired t ‑ test) when comparing metric scores across systems.
🧩 Connecting the Pieces
Text processing determines the quality of the inverted index; the index’s structure (dictionary + compressed
postings) dictates retrieval speed.
Analyzers bridge language characteristics with tokenisation, directly influencing term statistics that feed ranking
models.
PyLucene provides a Pythonic bridge to Lucene’s robust indexing/search engine, enabling rapid prototyping of
multilingual IR pipelines.
